# カレンダー下層ページ：痛みの根本原因分析

## 前提：この分析の問い

> ユーザーはなぜこのページに来るのか？
> 来た後、何を達成したいと思っているのか？
> 達成できなかったとき、彼らはどう感じるのか？
> それを解決することで、Zidookaは何を得られるのか？

## 1. ユーザーの到達文脈

### 月別ビュー（/2026/05）に来る人

**検索クエリの例**:
- `2026年5月 引越し 吉日`
- `2026年5月 大安`
- `2026年5月 一粒万倍日 いつ`
- `2026年5月 カレンダー 六曜`

**来る前の状態**: 具体的な予定（引越し、結婚、開業etc）があり、「日取りを決めなければならない」というタスクが発生している。多くの場合、親や上司から「縁起の良い日にしてほしい」と言われている。

**本当にやりたいこと**: カレンダーを見ることではない。**「予定を立てて、関係者に説明できる根拠を得る」**こと。

### 個別解説ページ（/taian など）に来る人

**検索クエリの例**:
- `大安 意味`
- `大安とは`
- `一粒万倍日 やってはいけないこと`
- `天赦日 結婚式`

**来る前の状態**: カレンダー上で「大安」と書いてあるのを見たが、意味がわからない。または、親から「大安の日にしなさい」と言われたが、なぜそれが良いのか説明できない。

**本当にやりたいこと**: 知識を得ることではない。**「大安にしろと言われたけど、本当に必要か？説得力のある判断材料が欲しい」**こと。

---

## 2. 感情マップ（Empathy Map）

### 月別ビューでユーザーが考えていること

| 段階 | 考えていること | 感じていること |
|---|---|---|
| **到着直後** | 「5月のカレンダーが見れる。どこが吉日かな」 | 期待、少しの不安 |
| **フィルタ使用前** | 「用途フィルタがある。引越しを選べばいいのかな」 | 迷い（正しい選び方がわからない） |
| **フィルタ使用後** | 「5月8日、15日、22日がハイライトされている。どれが一番？」 | 混乱（複数選択肢の比較負荷） |
| **日付クリック後** | 「大安と一粒万倍日が重なっている。これってすごいの？」 | 疑問（重複の価値が直感的にわからない） |
| **離脱時** | 「なんとなく15日が良さそう…」 | 不完全な満足、または諦め |

**核心の痛み**: 複数の「良い日」が提示されることで、**決定が逆に難しくなる**（選択肢のパラドックス）。

### 個別解説ページでユーザーが考えていること

| 段階 | 考えていること | 感じていること |
|---|---|---|
| **到着直後** | 「大安の説明ページ。読んでみよう」 | 好奇心 |
| **読み進め中** | 「大安は六曜の一つで…旧暦で…」 | 脱落（自分の状況と結びつかない） |
| **途中** | 「結婚式には向いている、と書いてある。でも私の場合は？」 | 距離感（一般論に留まっている） |
| **離脱時** | 「なんとなくわかった気がする…」 | 曖昧な理解、自信なし |

**核心の痛み**: 解説が**一般論に終始**しており、ユーザーの**具体例に結びつかない**。

## 3. 痛みの根本原因（5 Whys）

### 月別ビューのケース

**Q1**: なぜユーザーはカレンダーを見ても満足しないのか？
→ 複数の「良い日」があって、どれを選べばいいかわからない。

**Q2**: なぜどれを選べばいいかわからないのか？
→ 各日の「良さ」の相対的な優劣が示されていない。

**Q3**: なぜ優劣が示されていないのか？
→ カレンダーは「情報の平等提示」という設計思想になっている（すべての日を同じグリッドで表示）。

**Q4**: なぜ平等提示の設計になっているのか？
→ カレンダーというメディアの慣習に従っている（Googleカレンダー、iCalなどの影響）。

**Q5**: なぜカレンダーの慣習に従う必要があるのか？
→ ユーザーの「カレンダーを見る」という期待に応えるため。しかし、本当のユーザー行動は「日取りを決める」であり、「カレンダーを見る」は手段に過ぎない。

**根本原因**: **手段（カレンダー表示）が目的（日取り決定）を乗っ取っている**。

### 個別解説ページのケース

**Q1**: なぜ解説を読んでも自信が持てないのか？
→ 自分の状況に当てはまるかわからない。

**Q2**: なぜ自分の状況に当てはまるかわからないのか？
→ 解説が「定義→由来→一般的な意味」という百科事典的構造になっている。

**Q3**: なぜ百科事典的構造になっているのか？
→ SEO的に「大安 意味」という検索意図に応えるため。

**Q4**: なぜSEO的な構造がユーザーの満足につながらないのか？
→ 検索意図「大安 意味」は表面的であり、本質的な意図は「大安にすべきかどうか判断したい」である。

**Q5**: なぜ「判断したい」という本質が見えていないのか？
→ 検索クエリが「意味」という知識要求を投げかけているが、実際には**判断のための材料**が必要。

**根本原因**: **検索クエリの「言葉」に応えているが、検索クエリの「意図」に応えていない**。

## 4. 痛みから導かれる設計方向

### 方針：「減らす」とは何か

「増やす」の反対は「減らす」ではなく、「削ぎ落とす」である。

**削ぎ落とすべきもの**:
1. **ユーザーの判断負荷**（月別ビューで複数日を比較させない）
2. **百科事典的な前置き**（解説ページの定義・由来を最優先表示しない）
3. **情報の平等性**（すべての日を同じ大きさで表示する必要はない）

**残すべきもの**:
1. **決定力**（「この日が最適」という明確な推奨）
2. **具体性**（「あなたの場合は…」という文脈付き情報）
3. **信頼性**（「なぜこの日か」という説明の根拠）

## 5. 下層ページの再設計案（仮説）

### 月別ビュー（/2026/05）：「日取り相談」化

**現状の問題**: 30日分のグリッドが平等に表示され、ユーザーが比較・判断する負担が大きい。

**仮説**: フィルタ選択後、**推奨日を3つ以内に絞ってカード形式で提示**し、カレンダーグリッドは補助的に下に配置すると、決定が速くなる。

**具体案**:
```
【引越しに最適な日（2026年5月）】

★ 最適: 5月15日（木）
   大安 × 一粒万倍日
   理由：引越しに両方の吉日が重なる最も縁起の良い日

○ 良い: 5月8日（木）
   大安
   理由：六曜では最上級の吉日

△ 可: 5月22日（木）
   天赦日
   理由：天が赦す日。新しい生活の開始に適す

[カレンダー全体はこちら ▼]
```

**削ぎ落としているもの**: ユーザーが30日分を目で追って比較する作業
**残しているもの**: 判断の根拠と選択肢の階層化

### 個別解説ページ（/taian など）：「シチュエーション別ガイド」化

**現状の問題**: 「大安とは→由来→一般的な意味」という百科事典的構造。

**仮説**: ページの先頭に**「あなたはどの状況ですか？」**の選択肢を置き、選択後に状況別の解説を表示すると、ユーザーが自分ごととして受け止められる。

**具体案**:
```
【大安について知りたい状況を選んでください】

[結婚式・入籍を考えている] [引越しを考えている]
[開業・事業開始を考えている] [その他の予定がある]

↓ 選択後

「結婚式なら大安が最適な理由」
- 大安は「万事安泰」の意味
- 結婚という新しい家庭の始まりに、安泰の象徴が重なる
- ただし、大安だけでなく「天赦日」や「一粒万倍日」と
  重なる日があれば、さらに縁起が良い
- 2026年5月であれば、5月15日が大安×一粒万倍日
```

**削ぎ落としているもの**: 定義・由来など「知りたくてもない情報」の優先表示
**残しているもの**: ユーザーの文脈に即した情報

## 6. GA4カスタムイベント設計（痛み解決の検証）

### 測定の目的

以前のイベント設計は「何をクリックしたか」を測定していた。
今回は**「痛みが解消されたか」**を測定する。

### 痛み解消の代理指標（Proxy Metrics）

ユーザーの「満足」は直接測れない。代わりに、以下の行動を「満足の代理指標」とする:

| 代理指標 | 意味 | 測定イベント |
|---|---|---|
| **決定力** | ユーザーが特定の日付に「決めた」と感じた行動 | `calendar_date_confirmed` |
| **納得度** | ユーザーが納得して情報を他者に伝えたいと感じた行動 | `calendar_date_shared` |
| **理解度** | 解説を「自分ごと」として理解した行動 | `calendar_situation_selected` + `calendar_guide_completed` |
| **再訪意愿** | ユーザーがこのサービスを「頼りにする」と感じた行動 | `calendar_returned_within_7d`（7日以内再訪） |

### イベント詳細

#### `calendar_date_confirmed`

**発火条件**: 月別ビューで、1つの日付を開いた後、**他の日付を開かずに30秒以上経過**、または**ページ離脱**。

**理由**: 複数日を開き比較している = 迷っている。1日だけ開いてそのまま離脱 or 滞在 = おそらく決まった。

**パラメータ**:
- `date`: 開いた日付（YYYY-MM-DD）
- `luckyday_types`: その日付の吉日ラベル（配列）
- `purpose`: 選択中の用途フィルタ
- `comparison_count`: そのセッションで開いた日付の数（1なら比較なし）
- `time_on_detail`: 日付詳細を開いてからの滞在時間（秒）

#### `calendar_date_shared`

**発火条件**: 共有ボタンクリック

**パラメータ**:
- `share_type`: line/twitter/copy_url
- `date`: 共有した日付
- `source`: 共有元（月別ビュー/日付詳細/推奨カード）

**解釈**: 他者に伝えたい = 自信を持って決めた、または重要だと感じた。

#### `calendar_situation_selected`

**発火条件**: 個別解説ページでシチュエーション選択ボタンをクリック

**パラメータ**:
- `luckyday_type`: taian/ichiryumanbai/tensha など
- `situation`: marriage/moving/business/other

**解釈**: ユーザーが「自分の状況」を特定した = ページを「自分ごと」として受け止め始めた。

#### `calendar_guide_completed`

**発火条件**: シチュエーション別ガイドを最後まで読んだ（IntersectionObserverで75%以上のスクロール＋最下部のCTAが画面内に入った）

**パラメータ**:
- `luckyday_type`
- `situation`
- `time_on_guide`: ガイド表示後の滞在時間

#### `calendar_returned_within_7d`

**発火条件**: `calendar_date_confirmed` or `calendar_date_shared` を発火したユーザーが、7日以内にカレンダー関連ページに再訪問

**パラメータ**:
- `previous_date`: 前回確認/共有した日付
- `return_path`: 再訪したページパス

**解釈**: 7日以内に戻ってくる = 前回の訪問で信頼を得た。または、前回決めきれなかった。

### 否定的なシグナル（痛みが残っている証拠）

| イベント | 発火条件 | 意味 |
|---|---|---|
| `calendar_date_compared_3plus` | 1セッションで3つ以上の日付詳細を開いた | 比較しすぎ = 決められていない |
| `calendar_filter_switched_3plus` | 1セッションで3回以上フィルタを切り替えた | 用途が定まっていない、または迷っている |
| `calendar_bounce_under_10s` | ページ到達後10秒以内に離脱 | 期待と違った、またはすぐに答えが見つからなかった |
| `calendar_glossary_bounce_under_15s` | 解説ページ到達後15秒以内に離脱 | 百科事典的な前置きに耐えられなかった |

## 7. 仮説検証の流れ

### Phase 1: 現状計測（1-2週間）

現状のUIで上記イベントを計測し、ベースラインを取得:
- `calendar_date_confirmed` の発生率（sessionsに対する割合）
- `calendar_date_compared_3plus` の発生率
- 解説ページの `calendar_glossary_bounce_under_15s`

### Phase 2: 仮説検証（A/Bテスト or 時系列比較）

月別ビューに「推奨日カード」を追加:
- Treatment: フィルタ選択後、推奨日カードを最上部に表示
- Control: 現状のまま

測定:
- Treatmentの `calendar_date_confirmed` 率がControlより高いか？
- Treatmentの `calendar_date_compared_3plus` 率がControlより低いか？

個別解説ページに「シチュエーション選択」を追加:
- Treatment: ページ先頭にシチュエーション選択ボタンを配置
- Control: 現状の百科事典的構造

測定:
- Treatmentの `calendar_situation_selected` → `calendar_guide_completed` の到達率
- Treatmentの `calendar_glossary_bounce_under_15s` がControlより低いか？

### Phase 3: 学びの定着

検証結果から:
- 「推奨日カード」が効果的なら → 全月別ビューに展開
- 「シチュエーション選択」が効果的なら → 全個別解説ページに展開
- 効果がなければ → 仮説を修正（推奨の本数を変える、シチュエーションの分類を変えるなど）

## 8. Zidookaが得られるもの

### 直接的な価値

| 成果 | 測定指標 | ビジネス意味 |
|---|---|---|
| ユーザーが日取りを決められる | `calendar_date_confirmed` 率上昇 | 問題解決の実感 = ブランド信頼向上 |
| ユーザーが他者に共有する | `calendar_date_shared` 率上昇 | 口コミ発生 = 獲得コストゼロの流入 |
| ユーザーが再訪する | `calendar_returned_within_7d` 率上昇 | リピーター形成 = LTV向上 |

### 間接的な価値

- **信頼の構築**: 日取り決定という人生の重要イベントに関与することで、Zidookaは「縁起・吉日の専門家」としてのポジションを得る
- **データの蓄積**: どのシチュエーションでどの日が選ばれるかのデータが蓄積され、将来的な予測モデル構築に活用可能
- **収益化の前提**: 信頼があって初めて、関連サービス（引越し業者紹介、式場予約ツールetc）のアフィリエイトや提携が機能する

---

## 9. やらないこと（明確化）

この計画では以下を行わない:

1. **新しいページを増やさない**（既存の月別ビュー・解説ページの構造を変更するのみ）
2. **新しいコンテンツを量産しない**（既存のコンテンツの表示順序・構造を変えるのみ）
3. **新しい機能を追加しない**（フィルタやカレンダーグリッドはそのまま。表示方法を変えるのみ）
4. **広告を増やさない**（収益化は信頼構築後のフェーズで検討）

---

## まとめ

カレンダー下層ページの根本的な問題は「情報が多すぎて決定が難しい」こと。
解決策は「推奨を絞って提示し、判断負荷を減らす」こと。
GA4イベントは「痛みが解消されたか」を測定し、その仮説を検証するために使う。
これにより、Zidookaは「吉日カレンダー」から「日取りの相談相手」へ進化する。
